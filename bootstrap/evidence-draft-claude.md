## Invariants

- `[invariant]` Manifest schema v2 + event log v1 + state v1 are coupled; the engine derives state by replay. Evidence: `scripts/runtime/engine/src/derive-state.ts:56-292` and `schemas/event.schema.json:25-50`. Load-bearing because `deriveState(manifest, events)` is called as both a persistence step (`writeState`) and the canonical source of "where are we" for `resume`, `render`, and `dispatch-step`; any rewrite must preserve `f(events) -> state` as pure.
- `[invariant]` `events.ndjson` is append-only and the one true source of run progression. Evidence: `scripts/runtime/engine/src/append-event.ts:82-88` (appendFileSync only) and `derive-state.ts:41-51` (parse per-line). Ledger-over-artifacts is stated explicitly in `ARCHITECTURE.md:97-101` ("Ledger state takes precedence over artifact presence").
- `[invariant]` `circuit.manifest.yaml` is snapshotted into the run root at bootstrap and mutations are forbidden; on re-bootstrap the content must match exactly. Evidence: `scripts/runtime/engine/src/bootstrap.ts:128-138` throws on "manifest snapshot mismatch". Matters because state replay depends on a stable step/gate graph.
- `[invariant]` Attached run roots must live exactly at `<project>/.circuit/circuit-runs/<slug>`; enforced with realpath canonicalization. Evidence: `scripts/runtime/engine/src/bootstrap.ts:51-68` (`assertAttachedRunRoot`). Keeps continuity index relative paths meaningful and prevents cross-project attachment.
- `[invariant]` Two artifact tiers: canonical `artifacts/*.md` vs worker-private `phases/<step>/reports/*.md`. Parent circuits never promote worker reports directly. Evidence: `ARCHITECTURE.md:104-115` and `ARCHITECTURE.md:545-572` (workers-execute@v1 public vs private file list). Load-bearing because without this boundary, worker rewrites break parent circuits.
- `[invariant]` Gate kinds are a closed set: `schema_sections | all_outputs_present | checkpoint_selection | result_verdict | option_count`. Evidence: `schemas/circuit-manifest.schema.json:264-341` and `scripts/runtime/engine/src/runtime-core/types.ts:56-61`. New gate semantics require a schema bump.
- `[invariant]` `dispatch` steps MUST have `gate.kind = result_verdict` and `checkpoint` steps MUST have `gate.kind = checkpoint_selection`; enforced at schema validation time via conditional allOf. Evidence: `schemas/circuit-manifest.schema.json:143-191`. This is how "every dispatch is gated" is kept declarative.
- `[invariant]` Verdict vocabulary is globally enumerated (~35 verdicts) and per-protocol constrained. Evidence: `schemas/circuit-manifest.schema.json:347-387` (global enum) and `schemas/job-result.schema.json:119-536` (per-protocol allOf — e.g. `workers-execute@v1` must be `complete_and_hardened | issues_remain | coexistence_invalidated`). Adding a new protocol means adding a verdict whitelist here.
- `[invariant]` Route targets are a closed set: `@complete | @stop | @escalate | @handoff | <step-id>`. Evidence: `schemas/circuit-manifest.schema.json:343-346` and `runtime-core/types.ts:45-46`. The terminal targets are the only way to end a run.
- `[invariant]` Terminal statuses on run completion are typed tightly by `run_completed`'s terminal_target: `@complete -> completed`, `@stop -> stopped`, `@escalate -> blocked (+ diagnostic_path required)`, `@handoff -> handed_off (+ handoff_path required)`. Evidence: `schemas/event.schema.json:289-327`. Prevents unstructured termination.
- `[invariant]` Dispatch adapter resolution order is fixed: override → `dispatch.roles.<role>` → `dispatch.circuits.<circuit>` → `dispatch.default` → auto-detect (`codex` if installed, else `agent`). Evidence: `scripts/runtime/engine/src/dispatch.ts:221-243` and `README.md:205-210`. Receipt always reports `resolved_from` so debugging can reconstruct the choice.
- `[invariant]` Dispatch public roles are exactly `implementer | reviewer | researcher`. Evidence: `scripts/runtime/engine/src/dispatch.ts:16-18, 82-94`. Unknown role throws at config-parse time.
- `[invariant]` Built-in adapter names `agent`, `codex`, `codex-isolated` are reserved and cannot be shadowed by user configuration. Evidence: `scripts/runtime/engine/src/dispatch.ts:166-171`. Prevents users from overriding the isolated Codex launcher.

Evidence: `skills/workers/SKILL.md:72-87` + `scripts/runtime/engine/src/update-batch.ts:44-114`. ## Inferences

- `[inference]` Circuit is designed around the twin philosophies of "artifacts are state" and "events are history". Together they let a session crash at any point and still recover because the next session derives everything from events + snapshotted manifest. Basis: `derive-state.ts`, `resume.ts`, append-only `events.ndjson`, `circuit.manifest.yaml` snapshot rule. This is the foundational bet of v2.
- `[inference]` The verdict enum being globally-closed but per-protocol narrowed is an attempt to make worker outputs self-describing without letting protocols invent local vocabularies. Basis: the two-level enum in `circuit-manifest.schema.json` vs. `job-result.schema.json`. The tradeoff: adding a protocol is cheap in code but heavy in schema.
- `[inference]` The "runtime-core" directory plus `definition-ir` directory plus `catalog/` directory suggest circuit is mid-migration to a cleaner architecture but hasn't completed it. The classic runtime (`bootstrap.ts`, `derive-state.ts`, `dispatch-step.ts`) is what SKILL.md prose actually invokes via `.circuit/bin/circuit-engine`. Basis: CLI dispatch in `cli/circuit-engine.ts` imports from classic modules, not from `runtime-core/`. The newer layers look like scaffolding for a v3.
- `[inference]` The `/circuit:run` router is intentionally thin (one step, writes `active-run.md`, routes by prose rules in SKILL.md) because classification is a fuzzy task that benefits from the current session's context. Basis: `skills/run/circuit.yaml:22-37` is one synthesis step; all classification logic lives in `skills/run/SKILL.md` prose. This is a deliberate non-formalization.
- `[inference]` The workers adapter's separate batch state machine and separate `events.ndjson` exists because the inner implement-review-converge loop is finer-grained than the outer 6-step circuit graph; forcing it into the outer event schema would flood it with micro-events. Basis: `update-batch.ts` events are `attempt_started`, `impl_dispatched`, etc. — these would bloat the outer run's ledger. Cost: two parallel ledgers to reason about.
- `[inference]` The invocation ledger is not load-bearing for correctness but is load-bearing for debugging "why did this route here?". Basis: `appendLedgerEntry` is explicitly best-effort (returns bool, never throws). `invocation-ledger.ts:74-83`. Routing still works if the ledger fails; the operator just loses traceability.
- `[inference]` The `.circuit/bin/*` wrappers exist because SKILL.md prose needs stable paths that survive plugin cache moves. The wrappers are tiny (read `.circuit/plugin-root`, `exec "$PLUGIN_ROOT/...` on target), so the cost is negligible. Basis: `cli/user-prompt-submit.ts:120-162`.
- `[inference]` The "quiet by default" router behavior and the `--explain` flag suggest the team observed users wanting less chatter on common paths. Basis: `skills/run/SKILL.md:130-135`. This is a UX lesson embedded in code.
- `[inference]` The fact that adapter fallback is capped at one retry (explicitly commented) suggests there was a bug where repeated start-failures led to loops. Basis: `dispatch.ts:382-396` comment is an annotated review finding ("Review finding I5").
- `[inference]` The `codex-isolated` adapter and its PID janitor (`codex-runtime.ts`) exist because Codex CLI is invasive: it mutates `~/.codex/`, spawns child processes, leaves temp files. Isolating it is defensive. Basis: README.md:186-192 mentions "isolated CODEX_HOME and TMPDIR by default".

## Abstraction-by-Abstraction Map

**Workflow orchestration.** DOES: encode a stepwise DAG per workflow in `circuit.yaml` with `reads`/`writes`/`gate`/`routes`. Routes support normal next-step and terminal `@complete|@stop|@escalate|@handoff`. Resume is derived from replay. DOES NOT: support same-run workflow-to-workflow transfers (Build → Explore is explicitly not v1; the human must restart; `skills/build/SKILL.md:225-229`). HOTSPOT: `resume.ts:129-293` with its layered completeness logic (routes, artifact gate, jobs, checkpoints). QUOTE: "Build is the first workflow where the authored fixed graph and the outer runtime now meet on the real execution path" (`ARCHITECTURE.md:637-638`).

**Event-sourced state.** DOES: append-only `events.ndjson` + manifest snapshot + pure projection `deriveState`. Events are schema-validated with per-type conditional payloads (Ajv Draft 2020-12). DOES NOT: carry a concurrency control / CAS mechanism in the classic path (the `runtime-core/commit-ledger.ts` has `expectedRevision` + `expected_revision_mismatch` failure, which isn't used by the classic path). HOTSPOT: `derive-state.ts:56-292` — a 230-line switch statement. QUOTE: "Ledger state takes precedence over artifact presence. A step might have produced an artifact ... but resume must not advance from uncommitted local files" (`ARCHITECTURE.md:97-101`).

**Plugin surface.** DOES: exposes 9 public slash commands (`run, build, explore, repair, migrate, sweep, create, review, handoff`), hooks for `SessionStart` + `UserPromptSubmit`, and two shim dirs (`scripts/relay/` and hook-authored `.circuit/bin/`). Commands are generated from `catalog/prompt-surface-contracts.ts`. DOES NOT: expose MCP servers (none present in `.claude-plugin/plugin.json`); does not register slash commands dynamically beyond overlay manifest. HOTSPOT: the catalog-compiler system (regions with `<!-- BEGIN X -->` / `<!-- END X -->`) — powerful but anti-pattern to hand-edit.

**Config precedence.** DOES: upward-walk from cwd to git-toplevel, then `~/.claude/circuit.config.yaml`. First hit wins. Dispatch resolution layers on top: override → role → circuit → default → auto. DOES NOT: support environment variable overrides of adapter selection (only `CIRCUIT_PLUGIN_SKILL_DIR`, `CIRCUIT_PLUGIN_WORKERS_DIR`, `CIRCUIT_HOME` — all pre-dispatch). HOTSPOT: `dispatch.ts:96-199` parses config with explicit legacy-key-removal errors (mentions of `"roles"` at top level, `"commands"`, `"engine"`, `"per_circuit"`, `"per_step"`). That migration has already happened; the parse guards remain.

**Skill/tool registry.** DOES: `compose-prompt.sh --skills rust,tdd` appends `SKILL.md` of each named skill under `## Domain Guidance`. User config can set `circuits.<name>.skills`. DOES NOT: centralize a machine-readable skill inventory; does not validate skill budgets at compose time (budgets mentioned in `ARCHITECTURE.md:614-617` as "enforceable" but enforcement is prose in SKILL.md). HOTSPOT: resolve_skill is a linear scan of `SKILL_DIRS`.

**Cross-model prompting.** DOES: adapter dispatch supports (a) Claude Code Agent, (b) Codex CLI in isolated sandbox, (c) user-defined wrapper argv. All produce a uniform receipt JSON. DOES NOT: adapt the prompt per adapter; the prompt is identical except for an optional `<!-- dispatch-adapter: X -->` HTML comment hint (`compose-prompt.sh:315-320`). HOTSPOT: `codex-runtime.ts` ~hundreds of lines for isolation/cleanup. QUOTE: "The implement/review/converge loop, artifact chain, gates, and report format are identical regardless of adapter" (`skills/workers/SKILL.md:60-62`).

**Checkpoint / pause.** DOES: `kind: checkpoint` step with `checkpoint.options` + gate `checkpoint_selection`. Request/response JSON files written per-attempt. Rigor profiles gate whether the checkpoint actually blocks (skill prose decides). DOES NOT: support multi-question checkpoints or structured responses beyond `{selection}` (schema-constrained; `schemas/circuit-manifest.schema.json:250-262`). HOTSPOT: `checkpoint-step.ts` handles both request and resolve; prose in `SKILL.md` decides when to actually wait.

**Model/effort/skill-selection policy.** DOES: per-circuit skill override in config. Adapter resolution (which model runtime) via role/circuit/default. DOES NOT: support per-step skill/model selection in manifests (the `capabilities.required/optional` field in the step schema exists but I did not find it consumed); does not surface explicit model identifiers (workers run on whatever the adapter chooses). HOTSPOT: the gap between the schema's declarative `capabilities` and actual runtime behavior.

**Structured-output discipline.** DOES: JSON-schema validation of events, state, continuity records, circuit manifest, job results, surface manifest, invocation ledger. Report markdown sections enforced by prose + `schema_sections` gate. DOES NOT: use a typed intermediate like Zod at the TS boundary consistently — the classic path uses `as Record<string, any>` casts (`derive-state.ts:64-83`); the newer `runtime-core/` uses branded types. HOTSPOT: per-protocol verdict validation in `job-result.schema.json:281-534`.

**Custom-circuit authoring.** DOES: `/circuit:create` drafts to `~/.claude/circuit/drafts/<slug>/`, validates by bootstrapping through the engine, publishes to `~/.claude/circuit/skills/<slug>/`, materializes an overlay manifest. Reserved alias slugs (`fix, develop, decide, migrate, cleanup, overnight`) cannot be used. DOES NOT: hot-reload (requires `/reload-plugins`). HOTSPOT: the overlay/publish path touches `~/.claude/plugins/marketplaces/petekp` (`custom-circuits.ts:144-150`), coupling custom circuits to the petekp marketplace layout.

**Budget / rate limiting / observability.** DOES: invocation ledger for slash-command lifecycle; step `budgets.max_attempts` + `budgets.timeout_seconds`; dispatch receipts with `adapter/resolved_from/transport`; events.ndjson for run-level history; warnings surfaced on dispatch receipts. DOES NOT: enforce token budgets per prompt; no rate-limiting of dispatch; no metrics/tracing export. HOTSPOT: observability is fragmented across three distinct ndjson ledgers (per-run `events.ndjson`, per-batch `events.ndjson`, `~/.claude/circuit/invocation-ledger.ndjson`), each with its own schema.

## Hard Lessons for circuit-next

**Preserve by default.**
- `[inference]` Ledger + manifest snapshot + pure projection as the foundation of state ("artifacts are state, events are history"). Basis: Invariants on event log, manifest snapshot, derived state.
- `[inference]` Adapter boundary with receipt that names `adapter`, `transport`, `resolved_from`. Basis: Invariant on fixed adapter resolution order and seam on dispatch adapter boundary.
- `[inference]` Attached-run root enforcement (`.circuit/circuit-runs/<slug>`) via realpath canonicalization — protects continuity invariants. Basis: Invariant on attached-run enforcement.
- `[inference]` Closed gate-kind set with conditional schema validation (dispatch-must-have-verdict-gate, checkpoint-must-have-selection-gate). Basis: Invariants on gate kinds and dispatch/checkpoint gate coupling.
- `[inference]` Separation of canonical artifacts from worker-private relay state. Basis: Invariant on two-tier artifact model and seam on orchestrator↔worker contract.
- `[inference]` Per-project config overriding global config, with documented resolution order. Basis: Invariant on project-over-global precedence.
- `[inference]` `--invocation-id` threaded from hook to bootstrap for end-to-end traceability. Basis: Seam on invocation ledger ↔ bootstrap.

**Do NOT carry forward.**
- `[inference]` Two parallel runtime codebases (`classic` vs. `runtime-core`). Pick one architecture and commit. Basis: Failure mode on two runtime layers coexisting.
- `[inference]` The verdict enum bloat: 35 globally-declared verdicts + ~30 per-protocol conditionals in schema. A flatter data model (e.g., `{protocol, outcome, detail}` with per-protocol outcome types) would be easier to grow. Basis: Failure mode on open-but-closed verdict enum and per-protocol conditionals.
- `[inference]` Compiled-bundled CLIs committed to the repo with a "don't grep these" rule. Either commit source-only (with a build step on install) or commit small un-minified shims. Basis: Failure mode on bundled CLIs + `.rgignore`.
- `[inference]` Dual helper-shim paths (`scripts/relay/*.sh` AND `.circuit/bin/*`). Pick one. Basis: Failure mode on dual helper-shim paths.
- `[inference]` Per-protocol verdict conditionals in JSON Schema. Better expressed as TypeScript discriminated unions + a flat JSON schema with `verdict: string`. Basis: Failure mode on per-protocol verdict conditionals wall.
- `[inference]` The `run` skill having a `circuit.yaml` at all for its single-step "just write a markdown file" flow — it's a shim pretending to be a circuit. Basis: Fact on `skills/run/circuit.yaml` having one synthesis step.

**Re-examine with fresh eyes.**
- `[inference]` One-graph-per-workflow vs. a composable phase library. The shared phase spine suggests composability; the implementation duplicates it. Basis: Failure mode on duplicated frame-checkpoint pattern.
- `[inference]` Prose-as-program (SKILL.md files of 400-500 lines of bash+markdown that Claude follows literally). Works, but each edit is a risk surface. Basis: Failure mode on `build/SKILL.md` size and density.
- `[inference]` The workers adapter's inner state machine — is this a true second runtime, or can the outer runtime grow to model it natively? Basis: Failure mode on parallel batch state machine.
- `[inference]` `definition-ir` "v3-experimental" vs. the current v2 manifest. Is this abandoned, or the real target? Basis: Unknown on `definition-ir` integration status.
- `[inference]` Whether the adapter layer should know about *models* (sonnet, gpt, gemini) or only *transports* — current state is transport-only, but config suggests users want model awareness. Basis: Inference on transport-only adapter scope.
- `[inference]` Budget enforcement: manifest reserves `budgets.timeout_seconds` + `max_attempts` and capabilities exist, but runtime honoring is unclear. circuit-next could make this a first-class concern. Basis: Unknown on timeout enforcement and `capabilities` consumption.
